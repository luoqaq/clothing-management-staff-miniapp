## 当前有效约束
- 本项目定位为 ToB 店铺运营协同端，不引入 ToC 用户模型。
- 复用现有服装管理系统账号、商品、库存、订单、扫码与 COS 图片上传链路。
- 图片上传默认先在小程序端压缩，再走后端签发凭证直传 COS。
- 角色能力已和后台对齐：
  - 管理员可轻量上新
  - 销售及历史 `manager/staff` 统一按销售能力处理
  - 销售侧不展示供应商与成本价

## 视觉与交互状态
- 小程序已完成一轮全站视觉重设计，当前风格是轻奢服饰感路线。
- 全局设计 token、tabBar 图标、页面头部区、卡片区、操作区样式都已统一。
- 登录页不再预填默认账号密码，避免在门店终端暴露凭据。

## 联调与运行经验
- 小程序业务接口统一走 `/api/mobile`。
- 图片上传继续复用 `/api/assets/upload-policy`。
- 近期已修复一次微信开发者工具下的图片上传失败问题：
  - `chooseMedia` 返回的临时路径不能假定总能映射到 `wxfile://tmp/...`
  - 上传读取策略应保持“原始路径优先，转换路径兜底”
- 商品打印标签页的二维码现已改为小程序本地生成并直接绘制到标签 canvas：
  - 不再依赖外部 `api.qrserver.com` 下载二维码图片
  - 如果后续再出现真机间表现不一致，优先检查本地 canvas 绘制与保存链路，而不是外部网络可达性
- 若后续再遇到图片上传失败，优先检查临时文件路径、`getFileInfo/readFile` 读取策略和上传凭证申请时的文件名提取逻辑。

## 已知风险与待关注项
- 小程序图片压缩当前主要依赖质量压缩，未严格做尺寸重采样；如果后续要强控最长边，需要再补 canvas 缩放。
- 当前记忆不保留具体 AppID、线上瞬时状态、一次性构建结果；涉及现场联调时重新确认即可。
- 如果后端移动端接口继续扩展，优先保持与后台 web 端语义一致，避免录单与扫码链路分叉。

## 最近新增经验
- 订单列表展示顺序现在默认按 `createdAt desc + id desc` 兜底排序；即使连接到尚未升级稳定排序的后端版本，小程序端也会先做一次同口径排序，减少同秒多单时的展示抖动。
- 工作台与订单列表的订单数、销售额默认折叠，需点击展开，避免在门店场景下直接暴露经营数据。
